# 项目概览

## 1. 为什么做这个项目

这个项目的核心目标，不是尽快堆出很多代码，而是用一个真实可运行的 AI x Web3 系统，逐步学习并沉淀以下能力：

- Web3 基础
- Go / Rust 基础
- Solidity 基础
- AI 应用开发基础
- 后端系统设计基础
- 开源项目持续推进能力

项目希望最终具备三个价值：

1. 能跑起来，说明具备最小工程实现能力
2. 能持续演进，说明具备系统设计和长期推进能力
3. 能开源展示，说明具备结构化表达和项目沉淀能力

## 2. 项目目标

项目名称：

`AI-Driven Cross-Chain Risk & Intelligence System`

一句话说明：

构建一个基于链上数据索引、风险计算和 LLM 分析能力的学习型系统项目，逐步演化为可展示、可扩展的 AI x Web3 工程作品。

## 3. 用户当前起点

已有基础：

- 6 年软件开发经验
- 有 C++ / Qt 桌面开发背景
- 有一定工程能力
- 对底层、系统、高性能方向有兴趣

当前短板：

- Web3 基础不牢
- Go 不熟
- Rust 不熟
- Solidity 不熟
- AI 应用开发也不熟

因此本项目的设计原则必须是：

- 边学边做
- 只学当前任务真正依赖的知识
- 避免一开始进入过深理论
- 优先建立最小可运行闭环

## 4. 当前阶段划分

### 阶段 1：链上数据采集与入库

目标：

- 连接测试网 RPC / WebSocket
- 监听新区块
- 解析 ERC20 `Transfer` 事件
- 写入 PostgreSQL
- 提供基础查询 API

### 阶段 2：查询 API 与地址分析

目标：

- 增强查询能力
- 支持地址维度的数据分析
- 输出基础地址画像

### 阶段 3：风险规则与基础风控

目标：

- 建立初步风险规则
- 对地址或转账行为做规则判定

### 阶段 4：接入 LLM 做解释型分析

目标：

- 将规则结果和链上行为转成可读说明
- 尝试引入基础 AI 分析流程

### 阶段 5：增加 Agent / Tool Calling

目标：

- 探索更复杂的 AI 调度和工具调用

### 阶段 6：开源展示完善

目标：

- 完善 README、架构图、文档、演示说明

## 5. 当前阶段边界

当前只做阶段 1，不做以下内容：

- 多链支持
- 跨链风控
- AI Agent
- RAG
- 复杂规则引擎
- 高级性能优化
- 一开始就做大量抽象层

## 6. 当前建议技术选型

- Indexer：Go
- API：Go
- 数据库：PostgreSQL
- 网络：Sepolia

选择理由：

- Go 更适合当前快速建立最小闭环
- PostgreSQL 足够支持阶段 1 的结构化查询
- Sepolia 适合测试和学习，成本与风险更低

## 7. 当前成功标准

阶段 1 完成时，应至少满足：

- 可以稳定连接 Sepolia
- 可以解析并保存 ERC20 `Transfer` 事件
- 可以按地址查询转账记录
- 仓库具备基础文档、阶段记录和进度跟踪能力

## 8. 协作执行约束

后续在进入需要真实运行、连库、入库、起服务等阶段时，必须遵循下面的执行约束：

- 先确认当前代码应该落在哪个工程目录，而不是默认继续堆在之前的练习目录中
- 先确认本地环境是否具备运行条件，例如数据库、表结构、环境变量、依赖
- 如果前置条件不满足，优先把当前任务重定义为“环境准备任务”，而不是直接推进到运行验证
- 在每次准备进入“跑通闭环”之前，必须明确说明：
  - 当前工程位置
  - 当前环境状态
  - 当前缺失项
  - 当前真正可执行的下一步

特别约束：

- 不允许在未确认 PostgreSQL 已安装、可连接、数据表已创建的情况下，直接要求用户跑 Day 4 的完整入库闭环
- 不允许在未确认当前阶段工程目录的情况下，默认继续在旧的 Day 1 练习工程里扩展后续阶段代码

## 9. 阶段 1 正式工程位置

为避免后续 Day 2 ~ Day 5 的代码继续堆积在 Day 1 的练习目录中，阶段 1 正式工程位置明确为：

`ai-web3-risk-system/stage1-indexer-go/`

说明：

- `day1-go-rpc/` 保留为 Day 1 的最小练习样本
- `stage1-indexer-go/` 作为后续阶段 1 的正式工程目录，承载：
  - 日志识别与解析
  - PostgreSQL 连接与入库
  - 后续基础查询 API

## 10. PostgreSQL 最小准备步骤

在开始 Day 4 的真实入库验证前，至少要完成下面这些步骤：

1. 本地安装 PostgreSQL
2. 确认 PostgreSQL 服务可启动、可连接
3. 创建用于阶段 1 的数据库
4. 创建 `transfer_events` 表
5. 准备数据库连接配置，例如：
   - `PGHOST`
   - `PGPORT`
   - `PGUSER`
   - `PGPASSWORD`
   - `PGDATABASE`

执行原则：

- 如果以上任一步还没完成，就说明当前任务仍属于“环境准备阶段”
- 环境准备完成后，才进入“日志解析 -> 入库验证”的代码闭环

## 11. 教学与代码落地约束

后续教学必须遵循下面规则：

- 如果一个函数、结构体、SQL 或处理流程已经讲到足够具体，必须明确指出：
  - 这一轮是否应该把它写进代码
  - 应该写到哪个工程、哪个文件、哪个模块
- 如果当前还不适合写进代码，也必须明确说明原因，例如：
  - 前置环境未完成
  - 依赖未准备
  - 当前仍处于设计阶段
- 在代码构建过程中，必须主动检查环境依赖是否缺失，并把缺失项显式说出来
- 环境配置、依赖安装、数据库准备、本地运行验证，这些都属于项目学习内容，必须纳入教学流程
